# rdp Shell 开发路线计划

## 背景
本项目是 `rdp` (rewrite of rs-dash-pro)，一个严格遵循 POSIX.1-2024 标准的 shell 实现。目标是完全通过 Yash 测试套件，这是最严格的 POSIX 兼容性测试。

项目目前处于早期阶段：已建立完整的模块目录结构（`src/modules/`），但所有模块文件均为空或仅包含基础模块声明。现有两份详细的计划文档：
1. `PLAN/执行总纲.md` - 概念设计，阐述整体架构、设计哲学、IR 设计等
2. `PLAN/基础细节.md` - 详细实施计划，包含 7 阶段开发顺序

**重要原则**：本项目严格遵循 POSIX.1-2024 标准，绝不包含任何扩展语法（如局部变量、数组、进程替换等）。所有语法实现决策以通过 Yash 测试套件为唯一标准。

**注意**：一些非语法相关的增强功能（如行编辑、命令历史等）将在核心功能完成后考虑添加，这些不影响 POSIX 语法合规性。一切最终以`PLAN/执行总纲.md`和`PLAN/基础细节.md`为准.

## 设计原则
基于计划文档中的核心设计哲学：
1. **第一性原理思考**：从 POSIX 标准文本出发，而非模仿现有实现
2. **声明式优于命令式**：IR 层面采用声明-执行分离，便于优化
3. **尽早失败**：两阶段错误处理（编译期/运行时）
4. **平台无关核心**：所有平台相关代码隔离在抽象层
5. **所有语言元素一等公民**：特殊内置命令、复合命令等在 IR 中有直接对应
6. **名称修饰策略**：通过 `tmp_{index}_{varname}` 前缀实现变量隔离，避免子进程创建开销

## 总体开发策略
采用增量演进策略，确保每个阶段都有可工作的成果，并通过测试验证。

### 关键策略：
1. **骨架先行**：先建立所有模块的 NOP 实现，确保项目可编译运行
2. **自测驱动**：每实现一个模块，依据计划文档进行自测，验证是否符合设计
3. **Yash 测试贯穿**：随着功能完善，逐步运行 Yash 测试套件中的相关测试
4. **平台适配**：核心功能在 Linux 上验证后，通过平台抽象层适配 Windows

## 第0阶段：NOP 项目骨架搭建

### 目标
建立完整的模块结构，所有模块提供最小可编译的 NOP 实现，形成可运行的 shell 骨架。

### 工作内容
1. **lib.rs 和 main.rs**：定义模块结构，实现最基本的 shell 循环
2. **各模块的 mod.rs**：定义模块公共接口，提供空实现
3. **模块间依赖**：明确定义模块间的接口契约
4. **基础错误类型**：定义三层错误体系的基本类型
5. **基本构建与运行**：确保 `cargo run` 能启动 shell（显示提示符，接受输入但不执行）

### 预期成果
- 项目可编译运行
- 输入命令显示回显（或不处理）
- 模块结构清晰，为后续实现提供框架

## 第1阶段：核心基础设施

基于《基础细节.md》第十二章第一阶段，实现最基础的模块：

### 1.1 公共类型模块 (`src/modules/`)
- 定义核心类型：Token、AST 节点、IR 指令、值类型
- 定义接口契约：可展开、可优化、可执行、资源生命周期
- 建立类型系统基础

### 1.2 变量管理器 (`VarAndFuncManager`)
- 基本键值对存储（字符串类型）
- 支持设置、获取、删除
- 多表设计：普通表、导出表、只读表（基础框架）
- 名称修饰策略框架

### 1.3 平台抽象层核心 (`PlatformAbstraction`)
- Unix 实现：基本的进程创建/等待（`fork`/`exec`/`waitpid`）
- Windows 实现：`CreateProcess` 封装
- 统一接口：`spawn`、`wait`、`kill`、`pipe`
- 文件描述符管理框架

### 1.4 虚拟机基础架构 (`VMExecutor`)
- 寄存器文件
- 指令分派循环
- 基础指令：`CONST`、`MOVE`、`LOAD_VAR`、`STORE_VAR`
- 执行状态维护

### 1.5 错误处理基础 (`Errors`)
- 三层错误类型定义：编译期、运行时、组件内部
- 错误报告机制
- 位置信息跟踪

## 第2阶段：简单命令执行

### 目标
能够执行简单外部命令（如 `ls -l`）和内置命令（如 `echo`）。

### 工作内容
2.1 **词法分析器** (`Lexer`)
- 基本词法分析：单词、操作符、引用识别
- 别名展开框架
- 位置跟踪

2.2 **语法分析器** (`Parser`)
- 简单命令解析：命令名、参数、重定向
- 生成基础 AST

2.3 **IR 生成器** (`IRGenerator`)
- AST 到 IR 转换框架
- 简单命令的 IR 生成：`SPAWN_DEFINE`、`JOB_CREATE`、`JOB_START`、`JOB_WAIT`
- 变量访问指令：`LOAD_VAR`、`STORE_VAR`

2.4 **内置命令框架** (`Builtins`)
- `echo`、`cd`、`pwd` 等基本内置命令
- 内置命令调用机制

2.5 **参数解析器** (`Argparser`)
- 命令行参数解析
- 运行模式确定：交互式、脚本、`-c` 单命令

2.6 **REPL 基础** (`REPL`)
- 交互式循环
- 行编辑基础

### 验证标准
- 能执行 `ls -l` 并显示输出
- 能执行 `echo hello` 并输出 "hello"
- 能处理基本的重定向（如 `ls > file.txt`）

## 第3阶段：展开与引用

### 目标
实现 POSIX 定义的各个展开阶段。

### 工作内容
3.1 **参数展开** (`Expander`)
- `$var` 和 `${var}` 语法
- 默认值、替代值、错误处理：`${var:-default}` 等

3.2 **命令替换**
- `$(cmd)` 和反引号语法
- 名称修饰策略实现：`tmp_{index}_{varname}` 前缀
- 变量隔离机制，无需创建子进程或子shell
- 与 `(command)` 子shell语法共享相同的变量隔离机制

3.3 **算术展开**
- `$(( ))` 表达式解析
- 整数运算支持
- 类型系统：字符串到数值的转换

3.4 **波浪号展开**
- `~` 和 `~user` 展开
- 用户目录查找

3.5 **单词分割**
- IFS 变量支持
- 单词分割指令 `SPLIT_WORDS`
- 引用保护机制

3.6 **路径名展开**
- `*`、`?`、`[]` 通配符
- 文件系统匹配

### 验证标准
- 所有展开阶段按正确顺序执行
- 正确处理引用和转义
- 通过 Yash 展开相关测试

---

## 第4阶段：复合命令与控制流

### 目标
实现 POSIX 复合命令和控制结构。

### 工作内容
4.1 **if 语句**
- 条件判断
- then/else 分支
- 退出状态作为条件

4.2 **while/until 循环**
- 循环结构
- 条件判断
- `break`、`continue` 支持

4.3 **for 循环**
- 列表遍历
- 位置参数展开：`for i; do ...`

4.4 **case 语句**
- 模式匹配
- 通配符模式支持

4.5 **函数定义**
- 函数定义语法
- 位置参数处理
- 返回状态

4.6 **命令组合与名称修饰策略**
- `{ command; }` 命令列表与 `(command)` 子shell语法的语法区分
- 名称修饰策略实现变量隔离：`tmp_{index}_{varname}` 前缀
- 无需创建子进程，通过变量表前缀实现语义隔离
- 命令替换 (`$(cmd)` 和反引号) 采用相同机制

### 验证标准
- 所有复合命令正确执行
- 控制流逻辑正确
- 通过 Yash 复合命令测试

## 风险管理

### 技术风险
1. **POSIX 标准理解偏差**：通过 Yash 测试及早发现
2. **跨平台兼容性**：早期建立平台抽象层，定期双平台测试
3. **性能问题**：通过 SSA IR 和优化器逐步优化

### 项目管理风险
1. **范围蔓延**：严格限制为 POSIX 语法，绝不添加任何扩展语法（如局部变量、数组、进程替换等）。所有语法必须来自 POSIX 标准文本。非语法增强功能（行编辑、历史等）将在核心完成后考虑。
2. **进度延迟**：每个阶段都有可验证的产出，避免大瀑布式开发
3. **代码质量**：Rust 类型系统帮助保证安全，但需注意架构清晰度
