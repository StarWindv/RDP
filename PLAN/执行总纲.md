# Shell实现计划书: 概念设计 (详细版)

## 第零部分:

名称等名词不必完全按照此计划书设计, 可以有所出入, 本计划书只提供核心思想

## 第一部分: 项目概述与目标

### 1.1 项目定位

我们正在设计一个全新的命令解释器 (shell) , 它的核心目标是**完全遵循POSIX.1-2024标准**, 并通过业界公认最严格的POSIX兼容性测试套件——Yash(Yet Another Shell)测试套件。这意味着这个shell的行为必须与标准文本描述完全一致, 没有任何偏差。

### 1.2 为什么要做这件事？

现有的shell实现 (如bash、dash、zsh等) 要么包含大量扩展, 要么在某些边缘情况下与POSIX标准存在偏差。Yash测试套件之所以被称为“最严格”, 正是因为它会测试那些容易被忽略的边界情况——例如复杂的引号嵌套、展开顺序的精确性、信号处理的细节等。通过这个测试套件, 意味着我们的shell在这些细节上都做到了完美符合标准。

### 1.3 平台的考虑

这个shell需要能够在不同操作系统上运行, 主要包括Unix-like系统 (如Linux) 和Windows系统。关键点在于: 无论在哪个平台上运行, 它对用户展现的行为都必须是相同的POSIX语义。这意味着我们需要在Windows上模拟Unix的进程模型、信号机制等概念, 让POSIX脚本可以不加修改地运行。

### 1.4 不做什么

我们严格遵循POSIX标准, 不包含任何扩展特性。这意味着: 

- 不支持`[[ ]]`条件测试 (只用POSIX的`[`或`test`) 
- 不支持`local`变量 (所有变量全局共享) 
- 不支持数组
- 不支持`<( )`进程替换
- 等等

保持纯净的目的是确保兼容性, 扩展可以以后在“非POSIX模式”下添加, 但那是远期考虑。

## 第二部分: 整体处理流程

### 2.1 从输入到执行的旅程

当用户输入一条命令 (或运行一个脚本文件) 时, 它需要经过一系列处理阶段才能被执行。这些阶段是: 

```
原始输入 → 词法分析 → 语法分析 → 中间表示构建 → 优化 → 执行
```

每个阶段都是独立的转换步骤, 将输入逐步转化为更易于处理的形式。

### 2.2 各阶段做什么？

**词法分析**: 
把字符流切分成有意义的词法单元 (token) 。例如, 输入 `echo "hello world"` 会被切分成: 

- 关键字/命令名: `echo`
- 字符串: `"hello world"`
  这个阶段需要处理引号、注释、变量引用 (如`$var`) 、特殊参数 (如`$?`) 的识别。别名也在这一阶段被展开。

**语法分析**: 
根据POSIX定义的语法规则, 将词法单元组织成语法树 (AST) 。例如, `ls -l | grep txt` 会被解析成一个“管道”节点, 包含左右两个“简单命令”子节点。这个阶段要处理各种复合命令: `if`语句、`while`循环、`case`模式匹配、函数定义等。

**中间表示构建**: 
将语法树转换为一种更适合计算机分析和优化的内部形式。为什么需要这一步？因为语法树保留了太多源语言的细节, 不方便做优化。中间表示采用“静态单赋值”形式, 每个变量只赋值一次, 数据流关系非常清晰。我们后面会详细介绍中间表示的设计。

**优化**: 
在中间表示上应用各种等价变换, 使程序执行得更快或占用更少资源, 同时保证语义不变。例如, 如果发现一个变量被赋值为常量, 那么后续使用这个变量的地方可以直接替换为常量值 (常量传播) 。

**执行**: 
将中间表示转化为实际的操作系统动作。我们支持两种执行方式: 

- **解释执行**: 由一个虚拟机逐条执行中间指令
- **编译执行** (远期) : 将中间表示编译为机器码直接运行

这两种方式共享同一套中间表示, 所以行为完全一致。

## 第三部分: 语言元素的一等公民地位

### 3.1 什么是“一等公民”？

在传统shell实现中, 有些语言元素是被“特殊对待”的——它们在语法分析阶段就被识别, 而有些元素则被推迟到运行时处理。在我们的设计中, **所有POSIX定义的语言元素都在语法分析阶段被识别, 并在中间表示中有对应的结构**。这就是“一等公民”的含义。

### 3.2 复合命令

以下复合命令在中间表示中都有明确对应的控制结构, 而不是被展开成一系列原始命令: 

**if 语句**: 

```
if command1; then
    command2
else
    command3
fi
```

在中间表示中表现为条件分支结构, 包含条件表达式、then分支、else分支。

**while/until 循环**: 
有明确的循环结构, 包含循环条件和循环体。

**for 循环**: 

```
for i in 1 2 3; do
    echo $i
done
```

有明确的循环变量、值列表和循环体。

**case 模式匹配**: 

```
case $var in
    pattern1) cmd1 ;;
    pattern2) cmd2 ;;
esac
```

有明确的分支匹配结构。

**函数定义**: 

```
func() {
    commands
}
```

有明确的函数入口和出口。

### 3.3 特殊内置命令

以下命令被称为“特殊内置命令”, 因为它们在POSIX中有特殊的行为 (例如, 变量赋值在它们前面是持久的) 。这些命令在语法分析时就被识别, 直接生成对应的中间操作: 

**`: ` (冒号) **: 空命令, 总是返回成功。在中间表示中是一个空操作指令。

**`.` (点号) **: 读取并执行文件。相当于把文件内容插入当前位置执行。

**`break`**: 跳出当前循环。如果带数字参数 (如`break 2`) , 则跳出多层循环。这在中间表示中是一个跳转指令。

**`continue`**: 跳过当前循环迭代, 进入下一次迭代。也是一个跳转指令。

**`eval`**: 构造并执行命令。它的参数会被拼接成一个字符串, 然后作为shell代码重新解析执行。

**`exec`**: 替换当前shell进程为指定程序。如果带参数, 则用新程序替换shell；如果不带参数, 只修改重定向。

**`export`**: 将变量标记为需要传递给子进程。

**`readonly`**: 将变量标记为只读, 之后不能修改。

**`shift`**: 左移位置参数 (`$1`变成`$2`, 等等) 。

**`trap`**: 设置信号处理程序。

**`unset`**: 删除变量或函数。

为什么要把它们作为一等公民？因为这样可以在中间表示层面进行更精确的控制和优化。例如, `break`指令可以直接知道它要跳出哪个循环, 而不需要在运行时查找。

### 3.4 特殊参数

以下参数在词法分析时就被识别, 而不是作为普通变量处理: 

**位置参数**: `$1`、`$2`……`$9`, 以及`${10}`等。这些表示传递给脚本或函数的参数。

**特殊参数**: 

- `$@`: 所有位置参数, 每个参数单独加引号
- `$*`: 所有位置参数, 合并成一个字符串
- `$#`: 位置参数的个数
- `$?`: 上一个命令的退出状态码
- `$$`: 当前shell的进程ID
- `$!`: 最近一个后台命令的进程ID
- `$0`: 当前脚本的名称 (或shell的名称) 

它们在中间表示中有专门的读取指令, 与普通变量区分开。

### 3.5 测试命令的语法处理

`[`命令在POSIX中等价于`test`命令, 但有一个特殊要求: 它的最后一个参数必须是`]`。在我们的设计中, `[`在语法层面被特殊处理——解析器会确保它符合`[ 表达式 ]`的形式, 然后内部调用与`test`相同的逻辑。这样做的原因是: `[`可能不是系统中预装的程序 (尤其是在Windows上) , 把它作为内置语法可以保证它永远可用。

## 第四部分: 中间表示 (IR) 的详细设计

### 4.1 什么是中间表示？

中间表示 (Intermediate Representation, IR) 是介于源代码 (shell脚本) 和目标代码 (机器指令或虚拟机指令) 之间的一种程序表示形式。它既保留了足够的语义信息以便进行分析, 又足够底层以便进行优化和转换。

我们的IR采用**静态单赋值**形式。这意味着每个虚拟寄存器只被赋值一次 (在静态意义上) 。例如: 

```
%1 = 5
%2 = %1 + 3
%3 = %1 * %2
```

这里`%1`、`%2`、`%3`都是虚拟寄存器, 每个只被赋值一次。这种形式使得数据流关系非常清晰, 便于优化。

### 4.2 IR的核心设计原则

**原则1: 无限虚拟寄存器**
我们假设有无限多个虚拟寄存器可用, 每个寄存器可以存放一个值 (整数、字符串、资源标识符等) 。实际的物理寄存器分配由后端处理。

**原则2: 声明与执行分离**
这是IR设计中最重要的原则。我们区分两类指令: 

- **声明指令**: 创建资源对象, 但不产生实际的操作系统动作
- **执行指令**: 基于已创建的对象执行实际操作

这种分离使得优化器可以在实际执行前分析整个程序的结构。

**原则3: 显式表示所有POSIX概念**
POSIX标准中定义的所有概念——变量展开、单词分割、路径名展开、重定向、管道、作业控制等——都在IR中有对应的指令。没有任何隐含的运行时行为。

### 4.3 指令类别详解

#### 4.3.1 控制流指令

控制流指令改变程序的执行顺序: 

- **`BR %cond, %target1, %target2`**: 条件分支。如果`%cond`为真, 跳转到`%target1`, 否则跳转到`%target2`。
- **`JMP %target`**: 无条件跳转。
- **`PHI %dest, [%val1, %block1], [%val2, %block2]...`**: φ函数, 用于合并来自不同控制流路径的值。这是SSA形式必需的。
- **`CALL_FUNC %func_name`**: 调用函数。
- **`RETURN`**: 从函数返回。
- **`BREAK %levels`**: 跳出多层循环 (`%levels`指定层数) 。
- **`CONTINUE %levels`**: 继续下一次循环迭代。

#### 4.3.2 数据操作指令

- **`CONST %dest, value`**: 将常量值加载到寄存器。
- **`LOAD_VAR %dest, "varname"`**: 从变量表加载变量值到寄存器。
- **`STORE_VAR "varname", %src`**: 将寄存器值存储到变量表。
- **`LOAD_SPECIAL %dest, SPECIAL_?`**: 加载特殊参数 (如`$?`、`$$`等) 的值。
- **`MOVE %dest, %src`**: 寄存器间数据移动。

#### 4.3.3 展开指令

POSIX定义了一系列展开阶段, 每个阶段都有对应的指令: 

- **`EXPAND_TILDE %dest, %src`**: 波浪号展开。例如, `~/doc`变成`/home/username/doc`。
- **`EXPAND_PARAM %dest, %src`**: 参数展开。例如, `$var`变成变量的值。
- **`EXPAND_CMD %dest, %src`**: 命令替换。执行`%src`中的命令, 将其输出作为结果。
- **`EXPAND_ARITH %dest, %src`**: 算术展开。计算`%src`中的算术表达式。
- **`SPLIT_WORDS %dest, %src`**: 单词分割。根据IFS将字符串分割成多个单词。
- **`PATHNAME_EXPAND %dest, %src`**: 路径名展开 (通配符展开) 。例如, `*.txt`变成匹配的文件名列表。
- **`QUOTE_REMOVE %dest, %src`**: 引号删除。删除包围字符串的引号。

为什么需要这么多展开指令？因为POSIX规定这些展开必须按特定顺序执行, 并且有些展开 (如单词分割) 可能发生在其他展开之后。把每个展开都变成显式指令, 可以确保顺序正确, 也便于优化器分析。

#### 4.3.4 资源创建指令 (声明类)

这些指令创建各种资源对象, 但**不产生实际的操作系统调用**。它们只是创建“描述符”或“句柄”, 存放在寄存器中。

- **`%read_fd, %write_fd = CREATE_PIPE`**: 创建一个管道对象。返回两个值: 读端标识和写端标识。这两个标识可以传递给后续指令。

- **`%fd = OPEN_FILE %path, %mode`**: 打开一个文件。返回文件标识。`%mode`指定打开方式 (读、写、追加等) 。

- **`%proc = SPAWN_DEFINE`**: 定义一个进程模板。这个指令接受一个描述块, 包含: 
  
  - `executable`: 要执行的程序路径
  - `arguments`: 参数列表 (可以是寄存器中的值) 
  - `environment`: 环境变量 (可以是“继承”或指定的键值对) 
  - `stdin`: 标准输入来源 (可以是文件标识、管道读端、或“继承”) 
  - `stdout`: 标准输出去向
  - `stderr`: 标准错误去向
  - `cwd`: 工作目录
  - 其他属性
  
  例如: 
  
  ```
  %proc = SPAWN_DEFINE {
      executable: "/bin/ls",
      arguments: ["-l", "/tmp"],
      stdout: %pipe_write  // 输出到管道的写端
  }
  ```

- **`%job = JOB_CREATE %proc_list`**: 创建一个作业组, 将多个进程模板组合在一起。`%proc_list`是一个包含多个进程模板标识的列表。这些进程将作为一个整体被管理。

#### 4.3.5 执行指令 (动作类)

这些指令基于已创建的资源对象, 真正调用操作系统执行操作。

- **`JOB_START %job`**: 启动一个作业组。这是真正创建进程的时刻。VM会调用平台抽象层, 为作业组中的每个进程模板创建实际的操作系统进程, 并确保它们按照指定的方式连接 (如管道连接) 。
- **`JOB_WAIT %job`**: 等待作业组中的所有进程结束。
- **`JOB_KILL %job, %signal`**: 向作业组中的所有进程发送信号。
- **`JOB_FG %job`**: 将作业组切换到前台 (让它占用终端) 。
- **`JOB_BG %job`**: 将作业组切换到后台。
- **`CLOSE %fd`**: 关闭文件或管道。

#### 4.3.6 内置调用指令

- **`CALL_BUILTIN %result, BUILTIN_CD, %arg`**: 调用内置命令`cd`, 参数在`%arg`中, 结果 (退出状态) 存入`%result`。

内置命令在IR生成时就被确定, 所以可以直接携带操作码 (如`BUILTIN_CD`) , 不需要运行时查找。这也使得以下错误可以在IR构建阶段被静态捕获: 

- 调用不存在的内置命令
- 内置命令的参数数量错误 (如`cd`带两个参数) 
- `break`/`continue`不在循环中

### 4.4 一个完整的例子

让我们看一个简单的管道命令, 以及它如何被翻译成IR: 

**原始脚本**: 

```
ls -l | grep txt > output.txt
```

**对应的IR序列**: 

```
// 1. 创建资源 (声明阶段) 
%read, %write = CREATE_PIPE                     // 创建管道

%proc1 = SPAWN_DEFINE {                          // 定义第一个进程
    executable: "/bin/ls",
    arguments: ["-l"],
    stdout: %write                               // 输出到管道写端
}

%proc2 = SPAWN_DEFINE {                          // 定义第二个进程
    executable: "/bin/grep",
    arguments: ["txt"],
    stdin: %read,                                 // 从管道读端输入
    stdout: FILE("output.txt")                    // 输出到文件
}

%job = JOB_CREATE [%proc1, %proc2]               // 创建作业组, 组合两个进程

// 2. 执行 (动作阶段) 
JOB_START %job                                    // 启动作业组
JOB_WAIT %job                                      // 等待作业组结束

// 3. 清理 (可选) 
CLOSE %read
CLOSE %write
```

这种表示有几个优点: 

- **清晰**: 数据流 (管道连接) 一目了然
- **可优化**: 优化器可以在JOB_START之前分析整个作业结构
- **可静态检查**: 如果`/bin/ls`不存在, 可以在执行前发现 (如果路径是常量) 
- **平台无关**: IR不关心底层是用fork还是CreateProcess

## 第五部分: 资源与作业模型

### 5.1 什么是资源？

在IR层面, 我们把操作系统中的各种对象抽象为“资源”。资源有类型, 有标识符 (存放在寄存器中) , 可以在指令间传递。资源类型包括: 

**管道资源**: 
代表一个单向数据通道。创建管道时得到两个资源标识符: 读端和写端。这两个标识符可以被传递给SPAWN_DEFINE, 指定进程的输入输出来源。

**文件资源**: 
代表一个已打开的文件。创建方式可以是打开现有文件、创建新文件等。文件资源有访问模式 (读、写、追加) 。

**进程模板资源**: 
这不是操作系统进程, 而是一个“蓝图”——描述了将来要创建的进程应该是什么样子。包含程序路径、参数、环境、重定向等信息。创建进程模板不产生实际进程。

**作业组资源**: 
代表一组将要一起运行的进程模板。作业组资源包含一个进程模板列表, 以及一些元数据 (如前台/后台状态) 。创建作业组也不产生实际进程。

### 5.2 为什么需要作业组？

考虑管道命令 `cmd1 | cmd2 | cmd3`。这三个进程应该: 

- 同时运行
- 通过管道连接
- 作为一个整体被管理 (比如按下Ctrl+C时, 三个进程都应该收到SIGINT) 

作业组就是用来表示这种“进程集合”的概念。一个作业组中的进程会被同时创建, 放入同一个进程组 (操作系统概念) , 作为一个单元进行作业控制。

### 5.3 作业组的生命周期

**创建阶段**: 
通过`JOB_CREATE`将多个进程模板组合成一个作业组。此时还没有任何操作系统进程存在。

**启动阶段**: 
通过`JOB_START`真正创建进程。这时VM会: 

1. 为作业组中的每个进程模板调用平台抽象层创建实际进程
2. 确保进程间的连接 (管道、文件重定向) 正确建立
3. 将所有进程放入同一个进程组
4. 根据作业组的前台/后台属性, 决定是否让这个进程组占用终端

**运行阶段**: 
进程正在执行。可以通过以下指令控制作业组: 

- `JOB_STOP`: 挂起作业组 (发送SIGSTOP) 
- `JOB_CONT`: 继续被挂起的作业组
- `JOB_KILL`: 终止作业组

**等待阶段**: 
父shell等待作业组结束。`JOB_WAIT`会阻塞直到作业组中的所有进程都退出。

**结束阶段**: 
作业组中的所有进程都已退出, 资源被回收。

### 5.4 前台与后台

作业组有一个重要属性: 是前台运行还是后台运行。

**前台作业**: 

- 占用终端
- 接收终端产生的信号 (如Ctrl+C产生SIGINT) 
- 父shell等待它结束 (在交互式模式下, shell会阻塞直到前台作业完成) 

**后台作业**: 

- 不占用终端
- 不接收终端信号
- 父shell不等待 (立即提示下一个命令) 

在IR中, 这个属性可以在`JOB_CREATE`或`JOB_START`时指定, 或者通过`JOB_FG`/`JOB_BG`指令动态改变。

---

## 第六部分: 变量与作用域

### 6.1 变量存储

在纯POSIX模式下, 变量存储是一个简单的键值对表: 

- 键: 变量名 (字符串) 
- 值: 变量的值 (字符串) 

这个表是全局共享的——任何地方对变量的修改都会影响到整个shell。

### 6.2 变量引用与赋值

在IR层面, 变量通过`LOAD_VAR`和`STORE_VAR`指令访问: 

```
// 赋值 x=5
%1 = CONST 5
STORE_VAR "x", %1

// 读取 y=$x
%2 = LOAD_VAR "x"
STORE_VAR "y", %2
```

### 6.3 特殊参数的存储

特殊参数 (如`$?`、`$$`等) 不是存储在变量表中的。它们有专门的`LOAD_SPECIAL`指令: 

```
%status = LOAD_SPECIAL SPECIAL_?  // 读取 $?
%pid = LOAD_SPECIAL SPECIAL_$     // 读取 $$
```

### 6.4 函数与作用域 (远期扩展)

虽然纯POSIX没有局部变量, 但将来如果支持`local`扩展, 我们的变量模型可以扩展为作用域链: 

- 每个函数调用创建一个新的作用域帧
- 变量名按作用域修饰, 查找时沿作用域链向上
- `local`变量在当前帧创建, 非`local`变量在全局帧创建

但初期只实现全局变量。

---

## 第七部分: 执行模型与VM职责

### 7.1 VM的核心职责

虚拟机 (VM) 是执行IR指令的引擎。它的核心职责包括: 

**维护执行状态**: 

- 虚拟寄存器的当前值
- 变量表
- 已创建的资源对象 (管道、文件、进程模板、作业组) 
- 当前执行的指令位置
- 函数调用栈

**执行指令**: 
逐条解释执行IR指令。对于声明类指令 (CREATE_PIPE、SPAWN_DEFINE等) , 只在内部创建资源对象, 不调操作系统。对于执行类指令 (JOB_START等) , 调用平台抽象层完成实际操作。

**管理作业**: 

- 维护所有活动作业组的状态
- 处理作业控制命令 (fg、bg、jobs) 的语义
- 在作业组之间切换前台/后台

**信号处理**: 

- 接收来自操作系统的信号 (如SIGINT) 
- 根据信号和当前前台作业的状态, 决定如何响应
- 执行用户通过`trap`设置的信号处理程序

**错误处理**: 

- 处理指令执行中的错误 (如JOB_START失败) 
- 更新`$?`的值
- 根据`set -e`等选项决定是否终止执行

### 7.2 执行循环

VM的核心是一个简单的指令分派循环: 

```
while (还有指令要执行) {
    检查是否有信号到达 (如果有, 处理信号) 
    取出一条指令
    根据指令类型分派到对应的处理函数
    更新指令指针
}
```

这个循环在两种情况下会暂停: 

1. 遇到`JOB_WAIT`且等待的作业组在前台——这时VM会阻塞直到作业组结束
2. 脚本执行完毕

### 7.3 信号处理的实现

信号处理是一个关键且复杂的部分。在我们的设计中, 信号处理是这样工作的: 

1. 操作系统信号到达时, 被一个专门的信号处理器捕获
2. 信号处理器只做最少的工作: 设置一个“有信号到达”的标志, 可能通过一个自醒管道通知主循环
3. VM的主循环在每条指令执行前检查这个标志
4. 如果发现有信号, VM暂停当前指令执行, 处理信号: 
   - 如果信号对应某个`trap`命令, 执行trap中指定的命令
   - 如果是SIGINT且当前有前台作业, 将信号转发给前台作业组
   - 更新相关的shell状态

这种设计避免了在信号处理器中做复杂操作 (那很容易导致死锁) , 同时也保证了信号能得到及时处理。

### 7.4 IR与VM的边界

一个重要的设计原则: **IR描述“做什么”, VM负责“怎么做”**。

IR中描述: 

- 要创建什么资源 (管道、进程模板、作业组) 
- 资源之间如何连接 (哪个进程的输出去哪个管道) 
- 哪些进程属于同一个作业组

VM负责: 

- 将这些描述转化为具体的操作系统对象 (真正的管道、真正的进程) 
- 决定创建时机 (一次性启动整个作业组) 
- 维护运行时状态 (哪些进程在运行、它们的退出状态) 
- 处理与操作系统的交互 (等待、信号转发) 

这种分离使得IR可以保持纯净和平台无关, 而VM可以针对不同平台有不同的实现。

## 第八部分: 平台抽象层

### 8.1 为什么需要平台抽象层？

我们的shell需要在Linux和Windows上运行, 但这两个操作系统的底层机制完全不同: 

- 进程创建: Linux用fork+exec, Windows用CreateProcess
- 进程标识: Linux用PID, Windows用句柄和进程ID
- 信号: Linux有完整的POSIX信号, Windows没有信号概念
- 管道: Linux有匿名管道, Windows也有但语义略有不同
- 终端控制: Linux有复杂的termios, Windows有控制台API

平台抽象层的职责就是**向上层 (VM) 提供统一的POSIX语义接口, 隐藏底层的平台差异**。

### 8.2 抽象层提供的概念

**进程相关**: 

- `spawn(program, args, env, stdio_map)`: 创建并启动一个进程
- `wait(pid)`: 等待进程结束
- `kill(pid, signal)`: 向进程发送信号
- `getpid()`: 获取当前进程ID

**进程组相关**: 

- `setpgid(pid, pgid)`: 设置进程组
- `tcsetpgrp(fd, pgid)`: 设置前台进程组

**文件/管道相关**: 

- `pipe()`: 创建管道
- `open(path, flags, mode)`: 打开文件
- `close(fd)`: 关闭文件描述符
- `dup2(oldfd, newfd)`: 复制文件描述符

**信号相关**: 

- `sigaction(sig, handler)`: 设置信号处理器
- `sigprocmask(how, set)`: 屏蔽/解除屏蔽信号
- `sigsuspend(mask)`: 等待信号

**终端相关**: 

- `tcgetattr(fd, termios)`: 获取终端属性
- `tcsetattr(fd, termios)`: 设置终端属性

### 8.3 在Windows上模拟POSIX语义

对于Windows, 抽象层需要做大量的模拟工作: 

**模拟fork语义**: 
Windows没有fork, 但我们需要让用户感觉像是在fork。当VM要求创建一个进程时, 抽象层会: 

1. 收集所有需要继承的状态 (环境变量、文件描述符等) 
2. 调用CreateProcess创建新进程
3. 通过某种机制 (如命令行参数或共享内存) 将继承的状态传递给新进程
4. 新进程中的“启动代码”恢复这些状态, 然后执行目标程序

**模拟信号**: 
Windows没有信号, 但我们可以用事件和异常来模拟: 

- SIGINT: 可以用Ctrl+C事件模拟
- SIGTERM: 可以用TerminateProcess模拟
- SIGCHLD: 可以通过等待线程检测子进程退出

**模拟终端控制**: 
Windows控制台API与termios完全不同。需要封装: 

- 获取/设置控制台模式
- 管理控制台进程组
- 处理控制台事件 (Ctrl+C、Ctrl+Break等) 

### 8.4 抽象层的实现原则

1. **向上提供POSIX接口**: VM看到的总是POSIX风格的系统调用
2. **向下适配具体平台**: 内部根据平台调用不同的API
3. **语义第一, 性能第二**: 首先保证行为正确, 再考虑优化
4. **错误映射**: 将平台特有的错误码映射为POSIX定义的错误码

## 第九部分: 错误处理的两阶段模型

### 9.1 为什么分成两个阶段？

在传统shell中, 很多错误直到运行时才被发现。例如, 如果写错了内置命令名, 脚本会执行到那行才发现错误。这可能导致部分副作用已经发生 (比如创建了文件、修改了变量) 。

我们的设计将错误处理分为两个阶段: 

**阶段1: 编译期错误** - 在IR构建和优化阶段发现
**阶段2: 运行时错误** - 在执行阶段发现

这样可以尽早发现问题, 避免不必要的副作用。

### 9.2 编译期错误

这类错误在脚本实际执行前就能发现, 不产生任何副作用。包括: 

**语法错误**: 

- 缺少配对的引号
- 语法结构不完整 (如`if`没有`fi`) 
- 重定向符号后缺少文件名

**内置命令相关的静态错误**: 

- 调用不存在的内置命令
- 内置命令的参数数量错误 (如`cd`带了两个参数) 
- `break`或`continue`不在循环中

**某些可以静态确定的语义错误**: 

- 只读变量被赋值 (如果变量名是常量) 
- 函数重复定义

当发现编译期错误时, shell应该输出错误信息并退出, 不执行任何命令。

### 9.3 运行时错误

这类错误在执行过程中发生, 需要根据POSIX规则处理。包括: 

**命令执行错误**: 

- 外部程序不存在
- 权限不足
- 程序执行失败 (如加载器错误) 

**资源错误**: 

- 文件打开失败
- 管道创建失败
- 内存不足

**信号导致的中断**: 

- 用户按下Ctrl+C
- 收到SIGTERM

**脚本逻辑错误**: 

- 除零错误 (在算术展开中) 
- 变量未设置但`set -u`生效

### 9.4 运行时错误的处理规则

POSIX定义了一套复杂的规则来决定如何处理运行时错误: 

**命令退出状态**: 
每个命令执行后都会产生一个退出状态码 (0-255) 。这个状态码被存入`$?`。

**set -e (errexit) **: 
如果这个选项生效, 那么任何命令失败 (返回非零状态) 都会导致shell立即退出, 但有一些例外: 

- 失败的命令是条件判断的一部分 (如`if`的条件、`while`的条件) 
- 失败的命令是管道的一部分但管道整体成功
- 失败的命令被`!`反转

**set -u (nounset) **: 
如果这个选项生效, 引用未设置的变量会导致错误。

**trap 命令**: 
用户可以通过`trap`命令设置某些错误或信号的处理器, 在错误发生时执行特定命令。

VM需要跟踪当前执行上下文 (是否在条件中、是否在管道中、是否被`!`反转) , 以确定错误是否应该导致退出。

## 第十部分: 测试与验证

### 10.1 验证标准

项目的成功标准只有一个: **通过Yash测试套件**。

Yash是另一个POSIX shell, 它的作者编写了一套极其严格的测试套件, 用于验证shell是否符合POSIX标准。这套测试覆盖了标准的方方面面, 包括: 

- 词法分析的所有细节
- 语法解析的各种组合
- 每个展开阶段的精确顺序
- 内置命令的各种用法和边界条件
- 作业控制和信号处理
- 错误处理和退出状态

很多主流shell (包括bash的POSIX模式、dash) 都无法完全通过这套测试。

### 10.2 测试方法

Yash测试套件的使用方式很简单: 

```
yash run-test.sh /path/to/our/shell *.tst
```

每个`.tst`文件是一个测试用例, 包含要运行的命令和期望的输出/退出码。测试运行器会执行我们的shell, 运行测试脚本, 然后比较实际输出与期望输出。

### 10.3 测试策略

我们将采用增量测试策略: 

1. 先实现最基本的命令执行功能
2. 运行Yash中对应简单命令的测试, 直到全部通过
3. 逐步添加新功能 (管道、复合命令、变量展开等) 
4. 每次添加新功能后, 运行所有相关的测试, 确保没有破坏已有功能
5. 最终目标是所有测试全部通过

这种策略确保我们始终朝着正确的方向前进, 不会积累太多问题最后一起解决。

### 10.4 测试与平台

由于我们要支持多个平台, 测试必须在所有目标平台上运行。这意味着: 

- 对于每个平台, Yash测试套件都应该全部通过
- 如果某个测试在某个平台上失败, 说明平台抽象层有问题
- 某些测试可能涉及平台特有的行为 (如文件系统权限) , 需要确保抽象层正确模拟了POSIX语义

## 第十一部分: 总结与概念完整性回顾

### 11.1 设计哲学

这个shell的设计围绕几个核心哲学: 

**第一性原理思考**: 
我们不盲目照搬现有shell的实现方式, 而是从POSIX标准的文本出发, 思考“应该是什么”而不是“别人是怎么做的”。

**声明式优于命令式**: 
在IR层面, 我们倾向于描述“想要什么状态” (如管道连接、作业组) , 而不是“执行什么步骤” (如fork、dup2) 。这让优化和分析变得更加容易。

**尽早失败**: 
通过两阶段错误处理模型, 让许多错误在编译期就被发现, 减少副作用和调试困难。

**平台无关的核心**: 
核心的IR、优化器、执行逻辑完全与平台无关, 所有平台相关的内容都被隔离在抽象层中。

### 11.2 各层职责清晰

**词法/语法分析层**: 

- 输入: 字符流
- 输出: 语法树 (AST) 
- 职责: 确保输入符合POSIX语法

**IR构建层**: 

- 输入: 语法树
- 输出: SSA形式的IR
- 职责: 将高级语法结构转换为低级别指令, 所有语言元素都成为一等公民

**优化层**: 

- 输入: IR
- 输出: 优化后的IR
- 职责: 在保持语义不变的前提下提高执行效率

**执行层 (VM) **: 

- 输入: IR
- 输出: 操作系统动作
- 职责: 解释执行IR指令, 维护运行时状态, 调用平台抽象层

**平台抽象层**: 

- 输入: 高层操作请求 (如spawn) 
- 输出: 底层系统调用
- 职责: 将POSIX语义映射到具体平台, 隐藏平台差异

### 11.3 关键创新点

**声明-执行分离**: 
在IR层面明确区分资源创建和实际操作, 使得优化器可以提前分析程序结构。

**展开作为显式指令**: 
POSIX定义的每个展开阶段都有对应的IR指令, 确保顺序正确且可优化。

**进程模板和作业组**: 
将进程创建延迟到实际启动时, 允许在IR层面描述复杂的进程间关系。

**内置命令的静态处理**: 
在IR生成时就确定内置命令, 允许编译期检查错误, 减少运行时开销。

**两阶段错误处理**: 
将错误分为编译期和运行时, 尽早发现问题。

### 11.4 最终目标

通过这个设计, 我们期望得到一个: 

- **严格符合标准**: 通过Yash测试套件的shell
- **概念清晰**: 每一层都有明确的职责, 设计文档本身就是对POSIX shell的深入解读
- **跨平台**: 在Linux和Windows上提供一致的行为
- **可优化**: IR设计为优化留下了充足的空间
- **可扩展**: 未来可以在不破坏POSIX模式的前提下添加扩展功能

这个设计不仅是一个shell的实现计划, 也是对POSIX shell标准的现代化解读和概念重构。
