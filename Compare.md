# rs-dash 与标准 dash 功能对比分析

## 概述

本文档基于对 rs-dash（Rust实现的dash shell）和标准dash v0.5.3源代码的详细分析，对比两者之间的功能差异。分析旨在为 rs-dash-pro 项目的开发提供指导，目标是实现与POSIX标准的对齐。

## 架构对比

### 标准 dash 架构
- **解析器**: 基于Yacc的完整POSIX语法解析器 (`parser.y` → `parser.c`)
- **词法分析器**: 完整的词法分析 (`input.c`)
- **执行引擎**: 带完整作业控制的 fork-exec 模型
- **变量系统**: 带哈希表的复杂变量系统，支持属性（导出、只读等）
- **内存管理**: 自定义分配器（区域分配）
- **错误处理**: 集中式错误处理系统
- **作业控制**: 完整的进程组和会话管理
- **信号处理**: 所有POSIX信号处理

### rs-dash 架构
- **解析器**: 手写递归下降解析器，语法覆盖有限
- **词法分析器**: 与解析器集成，简单分词
- **执行引擎**: 简化的进程生成（`std::process::Command`）
- **变量系统**: 简单的 `HashMap<String, String>`
- **内存管理**: Rust标准分配器
- **错误处理**: 基本的错误打印
- **作业控制**: 完全缺失
- **信号处理**: 完全缺失

### rs-dash-pro 架构（增强版）
- **词法分析器**: 完整的POSIX Shell词法分析器 (`lexer.rs`)
- **解析器**: 递归下降解析器，生成AST (`parser.rs`)
- **SSA IR生成器**: 将AST转换为SSA IR形式 (`ssa_ir_generator.rs`)
- **优化器**: 空实现（计划中）
- **SSA执行器**: 执行SSA IR (`ssa_executor.rs`)
- **变量系统**: 基于 `ShellEnv` 的环境管理系统 (`env.rs`)

## 技术架构差异详情

### 1. 解析器架构对比

**标准 dash**:
- 基于Yacc的完整POSIX语法解析器
- 使用 `union node` 结构的解析树
- 所有shell构造的完整语法动作
- 支持行继续（`\` 后跟换行符）
- 完整的Here文档解析

**rs-dash**:
- 手写递归下降解析器
- 无适当解析树，直接执行
- 语法覆盖有限
- 不支持行继续
- 不支持Here文档

**rs-dash-pro**:
- 增强型递归下降解析器
- 生成AST结构
- 支持更多语法构造
- 支持行继续（在词法分析器中）
- 支持Here文档解析（在词法分析器中）

### 2. 执行模型对比

**标准 dash**:
- 完整的 fork-exec 模型
- 进程组和会话管理
- 全程信号处理
- 后台/前台作业跟踪
- 管道进程协调

**rs-dash**:
- 简化的进程生成（`std::process::Command`）
- 无作业控制或进程组
- 基本管道实现
- 无信号处理

**rs-dash-pro**:
- SSA IR执行模型
- 计划支持作业控制（未实现）
- 基本进程操作指令（Fork, Exec, Wait）
- 管道支持（在SSA IR中）

### 3. 变量系统对比

**标准 dash**:
- 带哈希表的复杂变量表（`vartab[VTABSIZE]`）
- 变量属性：导出、只读、整数、数组
- 函数的局部变量栈
- 子shell的变量继承
- 位置参数的特殊处理

**rs-dash**:
- 环境变量的简单 `HashMap<String, String>`
- 无变量属性
- 基本位置参数存储
- 无局部变量作用域

**rs-dash-pro**:
- `ShellEnv` 结构管理变量
- 支持变量查找和设置
- 基本的变量扩展
- 无完整的变量属性系统

### 4. 内置命令系统对比

**标准 dash**:
- 表驱动调度（`builtins.def.in` → 生成代码）
- 特殊内置命令（影响shell状态）
- 标准实用程序内置
- 执行的函数指针表

**rs-dash**:
- 带特征对象的注册表模式
- 每个内置命令实现 `Builtin` 特征
- 无特殊/赋值内置命令区分
- 手动注册

**rs-dash-pro**:
- 内置命令注册表系统
- 部分POSIX特殊内置命令已实现
- 命令执行通过SSA IR指令

### 5. 扩展系统对比

**标准 dash**:
- 复杂扩展管道（`expand.c`）
- 多个扩展阶段：波浪号、参数、命令替换、算术、路径名
- 控制字符（`CTLVAR`, `CTLBACKQ`, `CTLARI`）
- 上下文中的扩展（单词分割，引号删除）

**rs-dash**:
- `expansion.rs` 中的简化扩展
- 带有限上下文的单遍扩展
- 无控制字符表示
- 基本参数和命令替换

**rs-dash-pro**:
- 在SSA IR生成器中处理扩展
- 参数扩展作为SSA IR指令
- 命令替换作为SSA IR指令
- 无完整的波浪号扩展

### 6. 重定向系统对比

**标准 dash**:
- 复杂重定向处理（`redir.c`）
- 文件描述符操作
- Here文档处理
- 重定向排序和评估
- 重定向失败的错误处理

**rs-dash**:
- `redirection.rs` 中的基本重定向
- 简单的文件打开/dup操作
- 无Here文档支持
- 有限的错误处理

**rs-dash-pro**:
- 重定向作为SSA IR指令
- 支持多种重定向类型
- Here文档支持（在词法分析器中）
- 文件描述符操作指令

### 7. 错误处理对比

**标准 dash**:
- 集中式错误处理（`error.c`）
- `error()`, `warning()`, `syntaxerror()` 函数
- 交互模式的错误恢复
- 信号安全的错误报告

**rs-dash**:
- 带 `eprintln!()` 的基本错误打印
- 有限的错误恢复
- 无结构化错误类型

**rs-dash-pro**:
- 使用Rust的 `Result` 类型进行错误处理
- 解析错误和运行时错误分离
- 基本的错误恢复
- 无信号安全的错误处理

## rs-dash-pro 与POSIX标准的差距

### 核心POSIX合规性差距
1. **特殊内置命令**: 虽然已实现部分，但功能不完整
2. **作业控制**: 完全缺失，这是POSIX shell的核心功能
3. **信号处理**: `trap` 命令已定义但功能有限
4. **变量属性**: 缺少导出、只读、局部变量作用域
5. **扩展系统**: 参数扩展、算术扩展不完整

### 执行语义差距
1. **退出状态处理**: 需要更精确地模拟dash的退出状态语义
2. **错误处理**: 需要实现dash的 `set -e` 等错误处理选项
3. **命令查找**: 需要完整的PATH搜索和命令哈希
4. **脚本执行**: 需要处理shebang和脚本权限

### 交互功能差距
1. **历史系统**: 完全缺失
2. **行编辑**: 完全缺失
3. **补全功能**: 完全缺失
4. **作业控制交互**: 完全缺失

### 性能优化差距
1. **命令哈希**: 缺失
2. **快速路径执行**: 缺失
3. **字符串优化**: 基本实现
4. **内存管理优化**: 基本实现

## 总结

rs-dash 作为一个基础实现，提供了shell的核心功能，但距离完整的POSIX shell还有很大差距。rs-dash-pro 项目采用了更现代的SSA架构，为未来的扩展和优化提供了良好的基础。

主要差距集中在：
1. **POSIX特殊内置命令**: 虽然已实现部分，但功能不完整
2. **作业控制系统**: 完全缺失后台执行和信号处理
3. **变量系统**: 缺少属性、作用域和数组支持
4. **交互功能**: 缺少历史、行编辑和补全
5. **性能优化**: 缺少命令哈希等优化

对齐POSIX标准需要系统性地实现这些缺失功能，同时保持rs-dash-pro的SSA架构优势。SSA架构为优化提供了良好基础，但需要完整的执行语义实现才能发挥其优势。
