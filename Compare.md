# rs-dash 与标准 dash 功能对比分析

## 概述

本文档详细对比了 rs-dash（Rust实现的dash shell）与标准dash v0.5.3之间的功能差异。分析基于代码审查和测试，旨在为 rs-dash-pro 项目的开发提供指导。

## 架构对比

### 标准 dash 架构
- **解析器**: 基于Yacc的完整POSIX语法解析器 (`parser.y` → `parser.c`)
- **词法分析器**: 完整的词法分析 (`input.c`)
- **执行引擎**: 带完整作业控制的 fork-exec 模型
- **变量系统**: 带哈希表的复杂变量系统，支持属性（导出、只读等）
- **内存管理**: 自定义分配器（区域分配）
- **错误处理**: 集中式错误处理系统

### rs-dash 架构
- **解析器**: 手写递归下降解析器，语法覆盖有限
- **词法分析器**: 与解析器集成，简单分词
- **执行引擎**: 简化的进程生成（`std::process::Command`）
- **变量系统**: 简单的 `HashMap<String, String>`
- **内存管理**: Rust标准分配器
- **错误处理**: 基本的错误打印

## 技术架构差异详情

### 1. 解析器架构对比

**标准 dash**:
- 基于Yacc的完整POSIX语法解析器
- 使用 `union node` 结构的解析树
- 所有shell构造的完整语法动作
- 支持行继续（`\` 后跟换行符）
- 完整的Here文档解析

**rs-dash**:
- 手写递归下降解析器
- 无适当解析树，直接执行
- 语法覆盖有限
- 不支持行继续
- 不支持Here文档

### 2. 执行模型对比

**标准 dash**:
- 完整的 fork-exec 模型
- 进程组和会话管理
- 全程信号处理
- 后台/前台作业跟踪
- 管道进程协调

**rs-dash**:
- 简化的进程生成（`std::process::Command`）
- 无作业控制或进程组
- 基本管道实现
- 无信号处理

### 3. 变量系统对比

**标准 dash**:
- 带哈希表的复杂变量表（`vartab[VTABSIZE]`）
- 变量属性：导出、只读、整数、数组
- 函数的局部变量栈
- 子shell的变量继承
- 位置参数的特殊处理

**rs-dash**:
- 环境变量的简单 `HashMap<String, String>`
- 无变量属性
- 基本位置参数存储
- 无局部变量作用域

### 4. 内置命令系统对比

**标准 dash**:
- 表驱动调度（`builtins.def.in` → 生成代码）
- 特殊内置命令（影响shell状态）
- 标准实用程序内置
- 执行的函数指针表

**rs-dash**:
- 带特征对象的注册表模式
- 每个内置命令实现 `Builtin` 特征
- 无特殊/赋值内置命令区分
- 手动注册

### 5. 扩展系统对比

**标准 dash**:
- 复杂扩展管道（`expand.c`）
- 多个扩展阶段：波浪号、参数、命令替换、算术、路径名
- 控制字符（`CTLVAR`, `CTLBACKQ`, `CTLARI`）
- 上下文中的扩展（单词分割，引号删除）

**rs-dash**:
- `expansion.rs` 中的简化扩展
- 带有限上下文的单遍扩展
- 无控制字符表示
- 基本参数和命令替换

### 6. 重定向系统对比

**标准 dash**:
- 复杂重定向处理（`redir.c`）
- 文件描述符操作
- Here文档处理
- 重定向排序和评估
- 重定向失败的错误处理

**rs-dash**:
- `redirection.rs` 中的基本重定向
- 简单的文件打开/dup操作
- 无Here文档支持
- 有限的错误处理

### 7. 错误处理对比

**标准 dash**:
- 集中式错误处理（`error.c`）
- `error()`, `warning()`, `syntaxerror()` 函数
- 交互模式的错误恢复
- 信号安全的错误报告

**rs-dash**:
- 带 `eprintln!()` 的基本错误打印
- 有限的错误恢复
- 无结构化错误类型

## rs-dash-pro 项目状态分析

### 当前架构优势
1. **SSA架构**: 为优化提供了良好基础
2. **增强Lexer**: 支持完整的POSIX词法分析
3. **模块化设计**: 组件职责清晰
4. **跨平台支持**: Windows和Unix兼容

### 当前实现状态
1. **Lexer**: 完整POSIX token支持 ✓
2. **Parser**: 基本语法分析 ✓
3. **SSA IR生成器**: 基础功能 ✓
4. **SSA执行器**: 基本命令执行 ✓
5. **优化器**: 空实现 ✗

### 与标准dash的差距
1. **功能完整性**: 缺少大部分POSIX功能
2. **执行语义**: 缺少作业控制、信号处理等
3. **性能优化**: 缺少命令哈希等优化
4. **交互功能**: 缺少历史、行编辑等

## 总结

rs-dash 作为一个基础实现，提供了shell的核心功能，但距离完整的POSIX shell还有很大差距。rs-dash-pro 项目采用了更现代的SSA架构，为未来的扩展和优化提供了良好的基础。

主要差距集中在：
1. **POSIX特殊内置命令**: 缺失关键的内置命令
2. **作业控制系统**: 完全缺失后台执行和信号处理
3. **变量系统**: 缺少属性、作用域和数组支持
4. **交互功能**: 缺少历史、行编辑和补全
5. **性能优化**: 缺少命令哈希等优化

对齐POSIX标准需要系统性地实现这些缺失功能，同时保持rs-dash-pro的SSA架构优势。
